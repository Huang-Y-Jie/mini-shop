# 线上商城小程序开发文档（最终完整版·含实时订单+声音提醒）

## 一、项目概述

### 1.1 项目背景

本项目为基于小程序的线上商城系统，核心流程为「商品浏览→加入购物车→结算下单→支付完成」，后端采用FastAPI框架搭建，遵循RESTful API设计规范，集成小程序授权登录（JWT鉴权）与微信支付能力，购物车设计为「单用户单购物车」模式。针对支付场景的安全性需求，引入轻量级Redis缓存解决低并发下的订单幂等性、支付状态一致性问题；同时配套PC端管理后台，支持**订单实时展示+可选的新订单声音提醒**，满足商家高效运营需求。

### 1.2 核心功能

#### 小程序端（用户侧）

- 用户管理（小程序授权登录、地址维护）

- 商品展示（分类、详情、搜索）

- 购物车管理（添加、删除、修改数量、查询）

- 订单管理（购物车结算下单、订单查询、状态更新）

- 支付集成（线上支付、支付结果回调）

- JWT接口鉴权+Redis增强（Token主动失效、幂等控制）

#### PC管理端（运营侧）

- 管理员登录（独立JWT鉴权）

- 数据控制台（核心运营数据查看）

- 商品管理（新增、编辑、上下架、删除）

- 分类管理（新增、编辑、删除）

- 用户管理（列表查看、详情查看）

- 订单管理（**实时展示新订单**、列表筛选、详情查看、取消订单）

- 支付记录管理（对账、查询）

- 可选功能：**新订单电脑端声音提醒**

### 1.3 技术栈

|维度|技术选型|
|---|---|
|后端框架|FastAPI（Python）|
|数据库|MySQL（推荐8.0+）|
|缓存|Redis（5.0+，单节点）|
|实时通信|WebSocket（FastAPI实现）|
|认证方式|JWT（用户+管理员双套鉴权体系）+ 小程序授权登录|

|小程序前端|微信小程序原生开发/uni-app|

### 2.3 接口设计原则

- 遵循RESTful风格：资源为核心，使用HTTP方法表达操作（GET/POST/PUT/DELETE）

- 统一返回格式：包含状态码、提示信息、数据体

- 双端鉴权：用户端/管理端独立JWT Token，Token有效性通过Redis二次校验

- 支付/订单接口增加Redis幂等控制，防止重复操作

- 支付相关接口增加签名验证，防止参数篡改

- WebSocket连接需校验管理员Token有效性，确保安全

- 补充异常处理逻辑：明确Redis宕机、WebSocket推送失败、支付回调验签失败等场景的降级/补偿方案

- 定义核心错误码：明确409重复请求、401未授权等状态码的触发条件及自定义业务错误码

- 新增：商品下架/删除时主动清理购物车Redis缓存，避免脏数据；订单结算前强制校验商品状态

- 新增：订单超时未支付自动取消，通过Redis过期事件/定时任务实现，取消后推送状态至管理端

**购物车商品有效性校验逻辑**：

1. 基础逻辑：每次查询购物车时，自动过滤已下架/删除的商品，并同步更新Redis+DB；

3. 结算校验：订单确认页/创建订单前，强制校验商品状态，若存在已下架/删除商品，直接返回异常提示。

### 3.2 Redis缓存设计

|缓存Key|数据类型|过期时间|存储内容|用途|性能优化|
|---|---|---|---|---|---|
|`order:req:{req_id}`|String|5分钟|1（标记请求已处理）|订单创建幂等性控制|-|
|`order:status:{order_id}`|String|24小时|待支付/已支付/已取消/超时取消|支付状态缓存，减少DB查询|-|
|`user:token:{user_id}`|String|2小时|JWT Token值|用户Token主动失效控制|-|
|`admin:token:{admin_id}`|String|2小时|JWT Token值|管理员Token主动失效控制|-|
|`cart:{user_id}`|String|永久（同步DB）|购物车商品项JSON字符串|购物车缓存，提升响应速度|7天过期+惰性删除，防止内存溢出；商品下架/删除时主动清理|
|`product:detail:{product_id}`|String|30分钟|商品详情JSON字符串|商品详情缓存，减少DB查询|增加布隆过滤器防止缓存击穿|

|`order:timeout:{order_id}`|String|30分钟|订单ID|订单超时检测标记|监听Redis过期事件，触发订单自动取消|

## 四、接口设计（RESTful风格）

### 4.1 接口通用规范

#### 4.1.1 请求头

- 用户端鉴权：`Authorization: Bearer {user_token}`

- 管理端鉴权：`Authorization: Bearer {admin_token}`

- 数据格式：`Content-Type: application/json`

- 幂等性接口（创建订单）：`X-Request-Id: {唯一请求ID（UUID）}`

#### 4.1.2 统一返回格式

```JSON

{
  "code": 200,        // 状态码（200成功/400参数错误/401未授权/409重复请求/500服务器错误/503服务降级/410商品已下架）
  "msg": "success",   // 提示信息
  "data": {}          // 业务数据（可选）
}
```

#### 4.1.3 核心错误码定义

|错误码|含义|触发场景|
|---|---|---|
|200|成功|接口正常处理请求|
|400|参数错误|参数缺失/格式错误（如手机号、商品数量）|
|401|未授权|Token失效/不存在/权限不足|
|409|重复请求|相同req_id重复创建订单|
|410|商品已下架/删除|订单结算/创建时包含已下架/删除商品|
|500|服务器错误|后端逻辑异常/数据库操作失败|
|503|服务降级|Redis宕机等场景触发降级处理|

### 4.4 异常处理逻辑

|核心场景|异常处理逻辑|
|---|---|
|Redis宕机|1. 订单幂等性校验降级为DB唯一索引（order_no）；<br>2. 购物车查询/更新直接操作DB；<br>3. Token鉴权仅校验JWT本身有效性，放弃Redis二次校验；<br>4. 订单超时检测降级为定时任务扫描DB（expire_time）；<br>5. 记录降级日志，Redis恢复后自动同步缓存；<br>6. 接口返回503状态码，提示“服务临时降级”|
|WebSocket推送失败|1. 后端记录未推送成功的订单ID，启动定时重试（每10秒）；<br>2. 管理端前端定时拉取订单列表（每1分钟），兜底同步新订单；<br>3. 推送失败超过3次，触发管理员短信/微信提醒|
|支付回调验签失败|1. 记录异常日志（含回调参数、签名）；<br>2. 拒绝更新订单状态；<br>3. 人工核对后，提供后台手动更新订单状态的入口|
|接口限流触发|1. 订单创建/支付接口按IP+用户维度限流（单用户1分钟最多5笔订单）；<br>2. 触发限流后返回400状态码，提示“操作过于频繁，请稍后重试”|
|商品下架/删除后购物车脏数据|1. 主动清理Redis中包含该商品的cart:{user_id}缓存；<br>2. 用户查询购物车时强制过滤脏数据并同步DB；<br>3. 订单创建前校验商品状态，避免异常下单|
|订单超时未支付|1. Redis过期事件触发：更新订单状态为“超时取消”，同步Redis+DB，删除超时Key，WebSocket推送状态至管理端；<br>2. Redis宕机降级：定时任务（每1分钟）扫描DB中expire_time<当前时间且order_status=1的订单，批量更新状态；<br>3. 支付成功/手动取消时，删除Redis超时Key，终止超时检测|

## 六、实时订单+声音提醒实现逻辑

### 6.1 实时订单展示（WebSocket核心逻辑）

1. 管理员登录管理端后，前端自动发起WebSocket连接（`/ws/admin/order`），连接参数携带`admin_token`；

2. 后端校验`admin_token`有效性，验证通过后记录连接ID到Redis集合`admin:ws:clients`；

3. 用户创建订单成功后，后端完成订单入库+Redis缓存，设置订单超时Key（`order:timeout:{order_id}`，过期时间30分钟），随即遍历Redis中的在线管理员连接ID，广播新订单消息（包含订单ID、订单号、创建时间等核心信息）；

4. 管理端前端接收WebSocket消息后：

    - 更新订单列表（优先插入顶部/标红展示）；

    - 刷新控制台新订单统计数据；

    - 弹出新订单悬浮提醒框（可选）；

5. 订单超时取消/手动取消/支付成功后，后端广播订单状态更新消息，管理端同步更新列表；

6. 管理员退出/页面关闭时，前端主动关闭WebSocket连接，后端从Redis集合中移除对应连接ID；

7. 后端增加WebSocket心跳检测（每30秒），清理失效连接，避免资源浪费；

8. WebSocket推送失败时，触发降级机制（定时重试+前端兜底拉取）。

### 6.2 新订单声音提醒（可选功能）

1. 管理端提供全局开关（个人中心/订单列表页），开关状态通过`/api/admin/notify/setting`接口保存到Redis（`admin:notify:{admin_id}`）；

2. 前端预加载提醒音效文件（规范：mp3格式、大小≤100KB、时长≤3秒），确保播放无延迟；

3. 前端接收新订单WebSocket消息后，判断开关状态：

    - 开启：调用HTML5 Audio API播放提醒音效（循环1-2次，5分钟内多个新订单仅播放1次）；

    - 关闭：仅更新页面，不播放音效；

4. 音效播放优化：

    - 检测浏览器音频权限，未授权时给出“请开启音频权限”提示；

    - 静音设备下，仅展示视觉提醒（新订单标红+弹窗）；

    - 管理员手动查看新订单后，停止音效播放；

    - 订单超时取消/状态更新消息不触发音效。

### 6.3 商品下架/删除-购物车缓存清理逻辑（新增）

1. 管理员执行商品下架/删除操作时，后端接口（`/api/admin/product/{id}/status`/`/api/admin/product/{id}`）触发缓存清理流程：

    - 步骤1：查询DB中所有包含该商品的购物车记录（cart表），获取关联的user_id列表；

    - 步骤2：遍历user_id列表，删除Redis中对应的`cart:{user_id}`缓存Key；

    - 步骤3：记录缓存清理日志（商品ID、清理的用户数、操作时间）；

2. 用户再次查询购物车时，后端从DB拉取最新数据，自动过滤已下架/删除商品，并重新缓存到Redis；

3. 商品添加/修改状态时，前端下单/加购前强制校验商品状态，避免用户操作无效。

### 6.4 订单超时自动取消实现逻辑（新增）

#### 方案1：Redis过期事件监听（推荐）

1. 创建订单时，后端向Redis写入`order:timeout:{order_id}` Key，值为订单ID，过期时间30分钟；

2. 配置Redis开启过期事件通知（`notify-keyspace-events Ex`）；

3. 后端监听Redis的`__keyevent@0__:expired`事件，捕获`order:timeout:{order_id}` Key过期事件；

4. 触发过期事件后，后端执行：

    - 查询订单当前状态，仅处理“待支付”订单；

    - 更新订单状态为“超时取消”（order_status=4），同步更新Redis`order:status:{order_id}`缓存；

    - 广播WebSocket消息至管理端，通知订单超时取消；

    - 记录订单超时取消日志（订单ID、用户ID、取消时间）。

#### 方案2：定时任务降级（Redis宕机时）

1. 后端启动定时任务（每1分钟执行一次）；

2. 扫描DB中`expire_time < 当前时间`且`order_status=1`（待支付）的订单；

3. 批量更新订单状态为“超时取消”，同步Redis缓存；

4. 广播WebSocket消息至管理端；

5. 定时任务失败时，触发管理员告警（短信/微信）。

#### 终止超时检测的场景

1. 订单支付成功：支付回调接口删除`order:timeout:{order_id}` Key；

2. 手动取消订单：用户/管理员取消订单时，删除`order:timeout:{order_id}` Key；

3. 订单状态异常：定时任务/过期事件检测到订单非“待支付”状态时，直接终止。

### 6.5 Redis辅助支撑

- 在线客户端管理：通过Redis Set存储在线管理员WebSocket连接ID，支持快速遍历广播；

- 提醒开关状态：管理员声音提醒开关状态缓存到Redis（`admin:notify:{admin_id}`），读取更高效；

- 订单状态同步：新订单/支付状态更新/超时取消后，Redis缓存与WebSocket推送同步，保证数据一致性；

- 缓存防护：商品详情缓存增加布隆过滤器防止缓存击穿，购物车缓存设置7天过期+惰性删除，商品下架/删除时主动清理；

- 超时检测：通过Redis过期事件实现精准的订单超时取消，降级方案保障Redis宕机时功能可用。

## 七、Redis核心实现逻辑（基础功能）

### 7.1 订单幂等性控制

1. 前端生成唯一`req_id`（UUID），随创建订单请求传递；

2. 后端接收请求后，先查询Redis中`order:req:{req_id}`是否存在：

    - 存在：返回409错误（重复请求）；

    - 不存在：存入Redis并设置5分钟过期，继续创建订单逻辑；

3. Redis宕机时，降级为DB的order_no唯一索引校验幂等性；

4. 订单创建完成后，缓存订单状态到`order:status:{order_id}`，设置超时Key，触发WebSocket推送。

### 7.2 JWT Token主动失效

#### 用户端

1. 用户登录时，后端生成JWT Token，存入Redis（`user:token:{user_id}`），过期时间与Token一致；

2. 接口鉴权时，解析Token获取user_id，查询Redis中Token是否存在：

    - 不存在：返回401未授权；

    - 存在：正常处理请求；

3. 退出登录时，删除Redis中该user_id对应的Token缓存；

4. Redis宕机时，仅校验JWT签名有效性，放弃Token主动失效校验。

#### 管理端

1. 管理员登录时，后端生成JWT Token，存入Redis（`admin:token:{admin_id}`）；

2. 接口/WebSocket鉴权时校验Redis中Token有效性，退出时删除缓存；

3. Redis宕机时，仅校验JWT签名有效性。

### 7.3 购物车缓存策略

1. 读写策略：「先写Redis，后写DB」，查询时「先读Redis，未命中读DB并更新Redis」；

2. 主动清理：商品下架/删除时，后端主动遍历关联用户的`cart:{user_id}`缓存并删除，强制用户查询时拉取最新数据；

3. 数据同步：后端定时任务（每5分钟）将Redis购物车数据同步到DB，保证最终一致性；定时任务执行失败后自动重试（3次，间隔1分钟）；

4. 缓存失效：商品下架/删除时，主动清除对应商品的购物车缓存；

5. 商品有效性：

    - 查询购物车时自动过滤已下架/删除商品，同步更新Redis+DB；

    - 加购/结算/下单前强制校验商品状态，返回明确的异常提示；

6. 缓存防护：购物车缓存设置7天过期+惰性删除，防止内存溢出。

### 7.4 订单超时取消控制（新增）

1. 核心依赖：Redis过期事件+DB定时扫描降级；

2. 关键操作：

    - 创建订单：写入`order:timeout:{order_id}` Key，设置30分钟过期；

    - 支付成功/手动取消：删除`order:timeout:{order_id}` Key；

    - 过期触发：更新订单状态为“超时取消”，同步Redis/DB，推送WebSocket消息；

    - Redis宕机：定时任务扫描DB expire_time字段，批量处理超时订单；

3. 数据一致性：超时取消后，Redis`order:status:{order_id}`缓存与DB同步更新，避免用户查询到脏数据。

## 八、支付集成方案

### 8.1 微信支付完整流程

1. 前端结算下单：

    - 校验商品状态（已下架则提示无法购买）；

    - 生成`req_id`，传递选中的product_ids和`req_id`；

    - 后端Redis校验幂等性→创建订单（主表+订单项）→设置30分钟超时Key→缓存订单状态→触发WebSocket推送新订单；

2. 前端调用支付接口：传递order_id→后端调用微信支付API生成prepay_id等支付参数；

3. 前端通过`wx.requestPayment`调起微信支付；

4. 用户完成支付后，微信服务器回调后端`/api/pay/callback`接口；

5. 后端验证回调签名→删除超时Key→更新DB订单状态（主表+订单项）→同步更新Redis订单状态缓存→触发WebSocket推送订单状态更新；

6. 前端轮询「查询支付状态」接口（优先查Redis），感知支付成功后跳转订单详情页；

7. 管理端接收WebSocket状态更新消息，同步更新订单列表中对应订单的支付状态；

8. 订单超时未支付：Redis过期事件/定时任务触发自动取消→更新Redis/DB→推送WebSocket消息至管理端。

### 8.2 支付安全保障

- 幂等性：基于Redis的req_id防止重复下单、重复支付；支付回调基于微信交易号做幂等处理；

- 签名验证：支付回调接口验证微信支付签名，确保请求来源合法；

- 金额校验：订单金额=Σ（商品单价×数量），前后端双重校验；

- 状态一致性：DB与Redis同步更新订单支付状态，WebSocket推送保证管理端实时感知；

- 接口限流：支付/下单接口按IP+用户限流，防止恶意请求；

- 敏感数据保护：用户手机号、支付交易号等敏感数据脱敏展示，密码bcrypt加密存储；

- 超时控制：30分钟未支付自动取消订单，避免订单状态长期无效占用资源。

## 十、风险与注意事项

1. Redis数据安全：

    - 开启RDB+AOF持久化，防止服务器重启后幂等性Key、订单状态、在线客户端缓存、超时Key丢失；

    - 每日备份RDB文件至对象存储；

    - 开启Redis过期事件通知后，监控事件处理成功率，避免消息丢失；

2. WebSocket连接管理：

    - 增加心跳检测和失效连接清理，避免服务器连接数过载；

    - 推送失败触发降级重试机制；

    - 订单状态更新（含超时取消）需确保推送全覆盖；

3. 声音提醒体验：

    - 限制音效播放次数/时长（5分钟内仅1次），提供开关控制，检测音频权限/静音状态，避免影响管理员工作；

    - 仅新订单触发音效，订单状态更新/超时取消不触发；

4. 实时推送容错：

    - WebSocket推送失败时，管理端定时拉取最新订单（如每1分钟），保证订单不遗漏；

    - 超时取消订单需同时更新Redis/DB，避免数据不一致；

5. 管理端权限控制：

    - 仅超级管理员可操作商品/订单核心功能，WebSocket连接需校验管理员权限；

6. 支付回调幂等：

    - 基于微信交易号做幂等处理，防止重复更新订单状态；

7. 敏感数据保护：

    - 用户手机号、管理员密码等敏感数据加密存储，接口/WebSocket返回时脱敏；

8. 性能防护：

    - 商品详情缓存增加布隆过滤器，订单/支付接口限流，数据库索引优化，避免高并发下系统崩溃；

    - 商品下架/删除时批量清理购物车缓存，需控制批量操作频率（如分批次处理），避免阻塞主线程；

9. 多商品订单兼容：

    - 订单表拆分为主表+订单项表，支持多商品结算下单，避免单商品订单限制；

    - 结算前逐个校验商品状态，确保无已下架商品混入；

10. 购物车数据清洁：

    - 自动过滤已下架/删除商品，商品操作时主动清理缓存，定时同步Redis到DB，保证数据一致性；
